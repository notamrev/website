# Portfolio Site

A plain HTML/CSS portfolio site, generated by a small Node build script. No frontend framework.

- **Home** — short bio + a live, force-directed graph of GitHub activity (category → language → project, rooted at my GitHub account), built from `data/projects.json`
- **Newsletter** — learnings, synced from a Notion database at build time
- **Trips** — treks and other trips/personal projects, synced from a separate Notion database at build time, including photos
- **Projects** — public repos for [`notamrev`](https://github.com/notamrev) tagged with the `portfolio` GitHub topic, synced from the GitHub API at build time (being public isn't enough on its own — add the topic on a repo's page: gear icon next to "About" → Topics)
- **Resume** — the recruiter-facing page: contact links, skills, work experience, projects, education, from `data/resume.json`

Dark mode is a real theme (CSS custom properties swap, not a `filter: invert`), so photos in Trips render normally either way. Toggle button top-right, persisted in `localStorage`.

## How it works

```
data/resume.json      (hand-authored)   \
data/newsletter.json  (from Notion)      \
data/trips.json       (from Notion)       >  src/build.js  ->  dist/*.html
data/trip-images/     (downloaded once)  /
data/projects.json    (from GitHub)     /
```

`src/fetch-notion.js`, `src/fetch-trips.js`, and `src/fetch-github.js` each talk to a network API and write a JSON file under `data/`. `src/build.js` only reads those JSON files (plus `data/trip-images/`) and writes static HTML into `dist/` — it never touches the network. This means you can rerun `node src/build.js` alone to iterate on templates/CSS without re-hitting Notion or GitHub.

If a fetch script fails (bad token, unshared database, GitHub rate limit, network error), it logs a warning, leaves the existing `data/*.json` untouched, and exits 0 — the build still succeeds, with the affected page showing an empty-state message instead of crashing.

Trip photos are a special case: Notion's image URLs are signed and expire after about an hour, so `fetch-trips.js` downloads each photo once into `data/trip-images/<slug>/` (skipping ones already downloaded on a prior run) and rewrites the content to reference the local copy. `data/trip-images/` is committed to the repo, same as the other `data/*.json` files, so the site works from a fresh clone even without Notion access.

## Local setup

```bash
npm install
cp .env.example .env   # then fill in NOTION_TOKEN, NOTION_DATABASE_ID, NOTION_TRIPS_DATABASE_ID, see below
npm run build
npm run serve           # opens the site at http://localhost:3000
```

Useful variants:
- `npm run build:offline` — rebuild HTML from whatever's already in `data/*.json`, no network calls. Fast, good for template/CSS iteration.
- `npm run fetch:notion` / `npm run fetch:trips` / `npm run fetch:github` — rerun a single fetcher on its own.

## Notion setup

The newsletter content lives in a Notion database. Set it up once:

1. In Notion, create a new **full-page database** (e.g. titled "Newsletter" or "Learnings") with these properties:
   - `Title` (the default title property)
   - `Status` — Select, with options `Draft` / `Published` (only `Published` entries are synced)
   - `Date` — Date
   - `Tags` — Multi-select (optional)
   - `Summary` — Text (optional, shown on the newsletter index)
   - `Slug` — Text (optional; if omitted, a slug is derived from the title)
   - The page body (whatever blocks you write under each entry) becomes the post content.
2. Go to https://www.notion.so/my-integrations → **New integration** → name it (e.g. "portfolio-site-sync") → capability: **Read content** only → Submit.
3. Copy the **Internal Integration Secret** shown after creation — this is `NOTION_TOKEN`.
4. **Share the database with the integration**: open the database → "..." menu (or the "Connections" section of the share menu) → **Connect to** → select your integration.
   > This step is easy to miss — without it, the API returns 404 even with a valid token.
5. Get the **Database ID**: open the database as a full page, look at the URL: `https://www.notion.so/<workspace>/<DATABASE_ID>?v=...`. The `DATABASE_ID` is the 32-character string right before `?v=`.
6. Fill in `.env`:
   ```
   NOTION_TOKEN=secret_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   NOTION_DATABASE_ID=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   ```
7. Add at least one entry with `Status = Published` and some body content, then run `npm run build` to confirm it syncs.

Until this is set up, `npm run build` still works — the Newsletter page just shows "No posts yet."

### Trips database (same integration, second database)

Set up a second database for Trips, using the **same integration** (no need to create another one):

1. Create a new **full-page database** (e.g. titled "Trips") with these properties:
   - `Title` (the default title property)
   - `Status` — Select, with options `Draft` / `Published`
   - `Type` — Select, with options like `Trek`, `Road Trip`, `Project`, etc. (shown as a tag on each card; defaults to "Trip" if omitted)
   - `Date` — Date
   - `Location` — Text (optional, e.g. "Himachal Pradesh")
   - `Summary` — Text (optional, shown on the trips index and at the top of the post)
   - `Slug` — Text (optional; if omitted, derived from the title)
   - The page body is the trip write-up — add image blocks directly in Notion for photos, they get downloaded and embedded automatically.
2. Share this database with your existing integration the same way as the Newsletter one (step 4 above).
3. Get its Database ID the same way (step 5 above) and add it to `.env`:
   ```
   NOTION_TRIPS_DATABASE_ID=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   ```
4. Run `npm run build` — the first run will download every photo, which can take a bit for trips with lots of images; subsequent runs only download new ones.

## Deploying

Live at https://notamrev.github.io/website/, via `.github/workflows/deploy.yml`. It builds and deploys to GitHub Pages on:
- every push to `main`,
- manually via the Actions tab → "Run workflow", or
- a weekly schedule (Sunday 11:30 PM IST) — so new Notion content or newly-tagged `portfolio` repos go live even without a code change that week.

Repo secrets (`NOTION_TOKEN`, `NOTION_DATABASE_ID`, `NOTION_TRIPS_DATABASE_ID`) and Pages (Settings → Pages → Source: GitHub Actions) are already configured. Rotating a token or database ID just means updating the repo secret under Settings → Secrets and variables → Actions.

### Cheap custom domain

1. Buy a domain from a low-markup registrar — Porkbun, Cloudflare Registrar, or Namecheap are all reasonable; a `.dev`/`.me`/`.xyz` runs roughly $5–15/yr.
2. Add a `CNAME` file to the repo (or set the custom domain in the Pages UI, which creates it for you) containing your domain, e.g. `siddhantverma.dev`.
3. In your registrar's DNS settings, add either:
   - Four `A` records for the apex domain pointing at GitHub Pages: `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`, and/or
   - A `CNAME` record for `www` pointing at `notamrev.github.io`.
4. Back in repo Settings → Pages, enter the custom domain and enable **Enforce HTTPS** once DNS propagates (usually well under 24h).
5. Ongoing cost is just the domain renewal — GitHub Pages hosting itself is free for public repos.
